Hot-Swapping Credentials in a Docker Sandbox

How we solved credential expiry in Claude Code Docker sandboxes without destroying them.

Scroll to begin
Chapter 01

The Problem

AWS SSO credentials expire after a few hours. The Docker sandbox bakes environment variables in at creation time. When creds expire, Claude Code can't make Bedrock API calls.

The only documented fix? Destroy the sandbox and recreate it—losing session history, installed plugins, and workspace state.

We refused to accept that. This is the story of finding a better way.

Chapter 02

The Persistent Env File

Dead End

Write to the persistent env file

We noticed BASH_ENV=/etc/sandbox-persistent.sh and CLAUDE_ENV_FILE=/etc/sandbox-persistent.sh in the container's environment. These are hooks—any new bash shell sources that file on startup.

The idea: write fresh credentials there, and Claude would pick them up.

We wrote the creds. Verified they were in the file. And it didn't work. Claude kept failing with expired token errors.

Why it failed BASH_ENV only gets sourced when a new bash shell starts. The Claude Code process was already running with the old env vars baked into its process environment. There's no mechanism in Linux to change a running process's environment from outside.
Takeaway The file was a good idea for tools Claude spawns (bash commands), but not for Claude's own API calls.
Chapter 03

Kill and Reconnect

Dead End

Kill the Claude process, then reconnect

If the running process has stale env, kill it. When the user reconnects via docker sandbox run, the sandbox should spawn a new Claude process that sources sandbox-persistent.sh and gets fresh creds.

We killed both Claude processes (there were two—from separate connections). The container stayed alive because PID 1 is a watcher script, not Claude. We reconnected, and it still didn't work.

Why it failed docker sandbox run launches Claude via the container's CMD: claude --dangerously-skip-permissions. That's a direct exec of the binary—no bash involved, so BASH_ENV is never sourced. The new Claude process inherited the container's original environment (set by -e flags at creation time), not the file contents.
Takeaway The process starts clean from the container env, not from any shell profile. Writing to a file doesn't matter if nothing reads it.
Chapter 04

The Deep Recon

Two attempts down. Time to stop guessing and fully map the sandbox internals. We ran a systematic capability audit:

whoami User agent, uid 1000, in groups sudo and docker
sudo -l (ALL) NOPASSWD: ALL—Full passwordless root. This was the key discovery.
/proc/.../environ Readable but not writable. Can't directly modify the running process's env.
/proc/.../mem Writable (owned by agent). Could theoretically patch memory—incredibly fragile and complex.
nsenter Available. Could enter the process's namespace, but still can't change its env vars.
gdb / strace Not installed. No easy way to attach and inject env vars into the running process.
which claude This was the pivotal moment.
Chapter 05

The Pivotal Discovery

claude was a symlink:

agent@sandbox 
$ ls -la /home/agent/.local/bin/claude
/home/agent/.local/bin/claude -> /home/agent/.local/share/claude/versions/2.1.42

$ file /home/agent/.local/share/claude/versions/2.1.42
ELF 64-bit LSB executable # compiled binary, not a script

A compiled ELF binary. We couldn't edit it to add source logic.

But then the realization hit:

The Insight We don't need to modify the binary. We need to control what happens before the binary runs. The symlink is just a pointer. We can point it somewhere else.
Chapter 06

Shim the Symlink

Breakthrough

The solution

The logic was clean: the sandbox always invokes claude by name. It resolves to a symlink we can change. If we replace the target with a bash wrapper that sources the env file and then exec's the real binary, every future launch gets fresh creds.

Execution Flow — After the Shim
$ claude
→
wrapper.sh
→
source creds
→
exec claude.real

  1. Back up the real binary

    agent@sandbox
    $ sudo cp /home/agent/.local/share/claude/versions/2.1.42 \
       /home/agent/.local/share/claude/versions/2.1.42.real

  2. Write the wrapper script

    claude-wrapper
    #!/bin/bash
source /etc/sandbox-persistent.sh 2>/dev/null
exec /home/agent/.local/share/claude/versions/2.1.42.real "$@"

    Sources the credential file, then exec's the real binary. The wrapper disappears—Claude inherits the fresh env vars.

  3. Repoint the symlink

    before → after
    # Before:
claude -> /home/agent/.local/share/claude/versions/2.1.42  (ELF binary)

# After:
claude -> /home/agent/.local/bin/claude-wrapper  (bash shim)

  4. Kill the stale process, reconnect

    The running Claude still has stale creds—nothing can fix that. Kill it, reconnect, and the sandbox invokes claude → wrapper → fresh creds → real binary.

Chapter 07

Why It Works

Why exec matters

The wrapper uses exec, not a plain call. exec replaces the current process with the new one—the wrapper disappears and the real Claude binary takes over completely.

✓ No extra process lingering
✓ Claude doesn't know it was shimmed
✓ Process tree looks normal
✓ Signal handling works correctly

Three things that made it possible

If any of these were missing, the approach wouldn't work:

№

Passwordless Sudo

Needed to manipulate the symlink and binary owned by the install

≡

Writable Env File

/etc/sandbox-persistent.sh exists and is writable by agent

⊕

Symlink Resolution

Claude is invoked via a symlink we can redirect, not a hardcoded path

The sudo access was the most surprising—it's there because the sandbox is designed to be a throwaway environment where the agent needs to install packages freely.

Chapter 08

The Final Workflow

No sandbox destruction. Session history, installed packages, workspace state—all preserved. The refresh workflow from the host:

host machine — credential refresh 
# 1. Export fresh creds
eval "$(aws configure export-credentials --profile YOUR_PROFILE --format env)"

# 2. Write them into the sandbox
docker exec SANDBOX_ID bash -c "cat > /etc/sandbox-persistent.sh << EOF
export AWS_ACCESS_KEY_ID=\"$AWS_ACCESS_KEY_ID\"
export AWS_SECRET_ACCESS_KEY=\"$AWS_SECRET_ACCESS_KEY\"
export AWS_SESSION_TOKEN=\"$AWS_SESSION_TOKEN\"
EOF"

# 3. Kill the Claude process
docker exec SANDBOX_ID pkill claude
